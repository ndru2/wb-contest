# Динамическая симуляция логистики + AI-диспетчер

Система моделирует логистическую сеть (склады → хабы → ПВЗ) как динамический
ориентированный граф, предсказывает будущие перегрузки узлов с помощью **LightGBM**,
объясняет причины через **SHAP** и исполняет понятные правила **Decision Tree**:
объехать узел, сменить ПВЗ, поднять приоритет. Цель — заранее видеть перегрузку
и менять маршруты, пока пробка ещё не случилась.

## Что внутри

- `data/simulation.py` — графовая модель сети, заказы-агенты, динамическая стоимость рёбер.
- `src/data_loaders/loaders.py` — фичи для обучения и онлайн-инференса (одинаковый набор).
- `src/dispatcher/ai_dispatcher.py` — AI-диспетчер: LightGBM (риск/SHAP) + Decision Tree (правила).
- `src/dispatcher/closed_loop.py` — замкнутый цикл симуляции с диспетчером.
- `run_closed_loop.py` — обучение моделей и сравнение режимов в терминале.
- `viz_app.py` — интерактивная визуализация (Streamlit) «без диспетчера vs AI».

## Установка

Нужен Python 3.12. Из корня проекта:

```bash
python3 -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

> На macOS для LightGBM может потребоваться OpenMP: `brew install libomp`.

## Запуск

**1. Сравнение режимов в терминале** (обучение + метрики passive / reactive / ai):

```bash
source venv/bin/activate
python run_closed_loop.py
```

Выведет долю перегрузок, среднюю и пиковую загрузку, долю доставки по 3 режимам,
человекочитаемые правила дерева и SHAP-объяснение для узла в зоне риска.

**2. Интерактивная визуализация** (две панели рядом + анимация и правила):

```bash
source venv/bin/activate
streamlit run viz_app.py
```

Откроется в браузере. 🏭 склад · 🏢 хаб · 🏠 ПВЗ · 🚆 заказы на дорогах · 💥 перегрузка.
Кнопка ▶ запускает анимацию, ползунок перематывает по шагам.

> После изменения параметров перезапустите Streamlit или нажмите ⋮ → **Clear cache** → **Rerun**:
> тяжёлые расчёты кэшируются и сами не обновятся.

## Где менять параметры

Все ключевые настройки — вверху файла `run_closed_loop.py`. `viz_app.py` читает
их же (`NETWORK`, `REGIME`), поэтому достаточно править в одном месте.

### Сеть — `NETWORK` (`run_closed_loop.py`)

```python
NETWORK = dict(
    n_nodes=39, n_warehouses=5, n_hubs=14, n_pvz=20,   # СУММА должна сходиться!
    min_capacity_wh=60,  max_capacity_wh=100,          # ёмкость складов
    min_capacity_hub=10, max_capacity_hub=14,          # ёмкость хабов (узкое место)
    min_capacity_pvz=60, max_capacity_pvz=90,          # ёмкость ПВЗ
    warehouse_link_prob=0.6, hub_link_prob=0.8,        # плотность связей в слоях
    wh_hub_link_prob=0.9, hub_pvz_link_prob=0.9,       # плотность связей между слоями
)
```

- **Число узлов:** меняйте `n_warehouses`, `n_hubs`, `n_pvz`. Обязательно следите,
  чтобы `n_nodes == n_warehouses + n_hubs + n_pvz`, иначе будет ошибка.
- **Где будут перегрузки:** понижайте ёмкость нужного слоя. Сейчас узкое место —
  хабы (`min/max_capacity_hub`), потому что их диспетчер умеет объезжать. Если
  зажать ПВЗ — перегрузки уйдут на конечные точки, и объехать их будет нельзя.
- **Альтернативные маршруты:** чем выше `*_link_prob`, тем больше обходных путей
  у диспетчера (тем сильнее эффект AI).

### Поток заказов — `REGIME` (оценка) и `TRAIN_REGIME` (обучение)

```python
REGIME = dict(            # нагрузка при сравнении/визуализации
    min_orders_per_step=50, max_orders_per_step=90,   # сколько заказов в шаг
    hot_frac=0.3,                                      # доля «горячих» ПВЗ
    hot_multiplier=3.0,                                # во сколько раз выше спрос на них
)

TRAIN_REGIME = dict(      # отдельный, более тяжёлый режим ТОЛЬКО для обучения
    min_orders_per_step=60, max_orders_per_step=110,
    hot_frac=0.4, hot_multiplier=4.5,
)
```

- Больше спроса (`*_orders_per_step`) или сильнее горячие точки (`hot_*`) →
  больше перегрузок без AI → заметнее разница.
- `TRAIN_REGIME` держите тяжелее `REGIME`, чтобы модель видела достаточно
  перегрузок и училась на сбалансированных классах.

### Логика диспетчера

```python
N_STEPS = 150            # длина одного сценария при сравнении
RISK_THRESHOLD = 0.4     # порог P(overload), выше которого узел считается рисковым
PROACTIVE_FRACTION = 0.3 # доля рискового потока, объезжаемая проактивно (анти-стадность)
RISK_WEIGHT = 10.0       # насколько сильно прогноз риска влияет на стоимость рёбер (Layer 3)
EVAL_SEEDS = [1,2,3,4,5] # сценарии для усреднения метрик
TRAIN_SEED = 100         # seed генерации обучающих данных
TRAIN_STEPS = 500        # длина обучающего прогона
```

- **`RISK_THRESHOLD`** — ниже порог → диспетчер срабатывает раньше и чаще.
- **`RISK_WEIGHT`** — главный «рычаг» проактивности: чем больше, тем сильнее
  маршруты обходят узлы с высоким предсказанным риском (рабочий диапазон ~8–15).
- **`PROACTIVE_FRACTION`** — какую часть рискового потока переразложить заранее,
  чтобы все заказы не ринулись в один обход.

## Как читать метрики

Перегрузки возникают только на **хабах** (узкое место); склады и ПВЗ имеют запас
и почти всегда в норме, поэтому среднее «по всей сети» выглядит маленьким. Главные
показатели визуализации считаются по хабам: доля перегрузок хабов, средняя загрузка
хабов, пиковая загрузка. Там разница между «без AI» и «AI» хорошо видна.
